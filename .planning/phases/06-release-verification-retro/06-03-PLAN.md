<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user machine to GH Releases CDN | First-launch backend download crosses the public internet; SHA-256 verification defends against tamper |
| backend tar.gz to user filesystem | Extraction step writes hundreds of MB to APP_DATA_DIR — path traversal in tar entries is the classic risk |
| compile-time env var to runtime binary | OMNIVOICE_BACKEND_SHA256_* injected at build time; placeholder zeros mean dev builds skip verification |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06.03-01 | Tampering | Backend tar.gz served from a hostile mirror | mitigate | SHA-256 compile-time pin verified at runtime; mirror failure cascades to the next allow-listed mirror; allow-list only per Phase 3 INST-09 |
| T-06.03-02 | Tampering | Path traversal in tar entries | mitigate | Use the `tar` crate's safe extraction API (rejects `..` and absolute paths); never use raw shell `tar -xzf`; document this in the lib.rs implementation |
| T-06.03-03 | Tampering | Pill-mode widget silently broken by naive rebase | mitigate | This plan reimplements rather than rebases per Pitfall 3; checkpoint:human-verify Task 3 catches regression before tag |
| T-06.03-04 | Denial of Service | Mirror cascade fails on all mirrors, app cannot start | mitigate | Cancellation is graceful — app surfaces error toast with link to docs/install/troubleshooting.md and exits cleanly; user can manually download tar.gz and place it at APP_DATA_DIR/backend/ as a documented workaround |
| T-06.03-05 | Information Disclosure | Placeholder SHA (zeros) reaches a production build | mitigate | Tag-push workflow always sets the real env var via steps.backend-bundle.outputs.sha256; PR-event dry-runs are excluded from the publish path via 06-02's per-step guards; runtime WARN log flags any placeholder-SHA build |
| T-06.03-SC | Tampering | New crate adds (tar, flate2, sha2, possibly reqwest) | mitigate | Each added crate is pinned via Cargo.lock; verify versions are not premier-release (>= 1 month old) before merge; record audit results in the PR description; per Phase 0 Pitfall 4 — pin all GH Actions to SHA, applies analogously to Cargo.lock review |
</threat_model>
